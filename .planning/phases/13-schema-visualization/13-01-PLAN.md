    2. `_get_schema_metadata()` -- three pg_catalog queries:
       - Query 1 (tables): SELECT c.relname FROM pg_class c JOIN pg_namespace n WHERE n.nspname = 'public' AND c.relkind = 'r' ORDER BY c.relname
       - Query 2 (columns): SELECT table_name, column_name, data_type (via format_type), is_pk (via LEFT JOIN pg_constraint contype='p' with unnest(conkey)) -- filter attnum > 0 AND NOT attisdropped
       - Query 3 (foreign keys): SELECT source_table, source_column, target_table, target_column from pg_constraint contype='f' joined to pg_class and pg_attribute
       - Execute all three via `db.session.execute(query)` and return the three result sets
       - Use the exact SQL from RESEARCH.md Pattern 1 as the reference

    3. `_generate_mermaid_er(tables_data, columns_data, fk_data)`:
       - Group columns by table_name into a dict
       - Build fk_lookup dict mapping (source_table, source_column) to (target_table, target_column)
       - Output starts with "erDiagram" line
       - For each table: emit `    TABLE_NAME {` then for each column: `        SANITIZED_TYPE column_name MARKER` where MARKER is " PK" if is_pk, " FK" if in fk_lookup, else empty
       - After all entities: emit relationship lines using `TARGET ||--o{ SOURCE : "column_name"` per RESEARCH Pattern 2
       - Deduplicate relationships using a seen_rels set keyed on (source_table, target_table)
       - Per D-04: only column_name, type, and PK/FK markers -- no nullable/default/index info

    4. `_sanitize_pg_type(pg_type: str) -> str`:
       - TYPE_ALIASES dict mapping common PostgreSQL types to short Mermaid-safe tokens (per RESEARCH Code Examples: "character varying" -> "varchar", "timestamp without time zone" -> "timestamp", "timestamp with time zone" -> "timestamptz", "double precision" -> "float8", "boolean" -> "bool", "integer" -> "int", "bigint" -> "int8", "smallint" -> "int2")
       - Split on "(" to get base type, check TYPE_ALIASES
       - Fallback: replace spaces with underscores, strip parentheses and commas

    5. `_build_fk_adjacency_map(fk_data) -> dict`:
       - Build bidirectional adjacency: for each FK, add target to source's set and source to target's set
       - Convert sets to sorted lists for JSON serialization
       - Return the dict

    6. `_render_schema_diagram(mermaid_definition, fk_map, table_count, fk_count)`:
       - Returns f-string HTML fragment per UI-SPEC and RESEARCH Pattern 3
       - Include sr-only paragraph: "Database schema with {table_count} tables and {fk_count} relationships"
       - Container div: id="schema-diagram-container", class="overflow-auto bg-white p-4", style="min-height: 500px" (per D-07 and UI-SPEC)
       - Pre tag: class="mermaid" containing the mermaid_definition
       - Script block that calls mermaid.run() if mermaid is defined, then setTimeout(initSchemaInteraction, 500)
       - Inject fk_map as a JSON script block: `<script>window.__schemaFkMap = {json.dumps(fk_map)};</script>`
       - Handle empty state (zero tables): return the empty state markup from UI-SPEC (fas fa-database icon, "No tables found", body text)

    Discretion decisions (per CONTEXT Claude's Discretion):
    - Caching: Generate fresh on each request -- pg_catalog queries on ~30 tables execute in <10ms, caching adds invalidation complexity with no benefit
    - Table grouping: No grouping -- let Mermaid auto-layout handle it for ~30 tables
    - Mermaid config: securityLevel 'strict' (not 'loose') per RESEARCH Pitfall 3 -- ER diagrams lack native click callbacks so 'loose' adds XSS risk with no benefit

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/database/schema | Admin-only endpoint exposing database metadata |
| pg_catalog -> Mermaid text | Database metadata converted to diagram markup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Information Disclosure | /api/database/schema | mitigate | Endpoint protected by `@require_role("admin")` decorator -- only authenticated admin users can access schema metadata |
| T-13-02 | Tampering | Mermaid text rendering | mitigate | `securityLevel: 'strict'` in Mermaid config prevents script injection via diagram text; table/column names from pg_catalog are system-generated, not user input |
| T-13-03 | Spoofing | /api/database/schema | mitigate | `@require_role("admin")` requires authenticated Azure AD session via `@auth_required` chain |
| T-13-04 | Denial of Service | pg_catalog queries | accept | Queries scan ~30 tables in metadata catalog, execution <10ms; no user-controlled parameters that could amplify query cost |
| T-13-SC | Tampering | CDN dependency (Mermaid.js) | accept | Mermaid.js loaded from cdn.jsdelivr.net which is already in CSP allowlist; version pinned in CDN URL to prevent supply chain drift |
</threat_model>
