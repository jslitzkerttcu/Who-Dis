## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`p-1`) | Icon gaps, inline padding |
| sm | 8px (`p-2`, `gap-2`) | Compact element spacing, checkbox-to-label gap |
| md | 16px (`p-4`, `gap-4`) | Default element spacing, card padding |
| lg | 24px (`p-6`, `gap-6`) | Section padding, KPI card internal padding |
| xl | 32px (`gap-8`) | Layout gaps between dashboard sections |
| 2xl | 48px (`py-12`) | Major section breaks |
| 3xl | 64px (`py-16`) | Page-level spacing (unused in this phase) |

Exceptions: Checklist item rows use `py-3` (12px) for compact vertical density matching existing table row patterns.

## Component Inventory

### Page: Workflow Dashboard (`/admin/workflows`)

**Layout:** Admin sub-section page extending `base.html`. Follows the Phase 8 Reports pattern with KPI cards at top and tabbed content below.

**Navigation placement (D-12 decision):** Admin sub-section at `/admin/workflows`. Added as a new card on the admin index page (`admin/index.html`) alongside Compliance and Reports. Rationale: workflows are admin-only and the existing admin index card grid is the established pattern for admin features.

**KPI Cards (D-13 decision):** 4 KPI cards matching the Phase 8 license report pattern:

| Card | Icon | Value | Subtitle |
|------|------|-------|----------|
| Active Workflows | `fa-list-check` (blue-600) | Count | Currently in progress |
| Overdue Items | `fa-clock` (yellow-600) | Count | Past due date |
| Completed This Month | `fa-circle-check` (green-600) | Count | Finished workflows |
| Avg. Completion Time | `fa-stopwatch` (blue-600) | N days | Average days to complete |

KPI card structure:
```
bg-white rounded-lg shadow-sm border border-gray-200 p-6
  Row: text-xs font-normal text-gray-500 uppercase tracking-wider + icon
  text-2xl font-semibold text-gray-900 (value)
  text-sm text-gray-500 mt-1 (subtitle)
```

**Tab Navigation (D-14 decision):** Two tabs -- "Active" (default) and "Completed". Matches Phase 8 tab pattern with `border-b-2 border-blue-600 text-blue-600 font-semibold` for active tab. HTMX-powered tab switching with `hx-push-url`.

**Active Workflows Table:**

| Column | Content | Sort |
|--------|---------|------|
| Employee | Name + email | text |
| Type | Onboarding / Offboarding badge | text |
| Job Code | Code + title | text |
| Progress | N/M items (progress bar) | number |
| Overdue | Count (red if > 0) | number |
| Created | Date + "by [admin]" | date |
| Actions | View button | -- |

Progress bar: `bg-gray-200 rounded-full h-2` container, `bg-blue-600 rounded-full h-2` fill, `bg-green-600` when 100%.

Overdue badge: `bg-red-100 text-red-800 text-xs font-medium px-2.5 py-0.5 rounded-full` when count > 0.

Type badge: Onboarding = `bg-blue-100 text-blue-800 text-xs font-medium px-2.5 py-0.5 rounded-full`. Offboarding = `bg-orange-100 text-orange-800 text-xs font-medium px-2.5 py-0.5 rounded-full`.

**Completed Workflows Table:** Same columns minus Overdue, plus "Completed" date column. Paginated using Phase 1 `paginate()` helper.

### Page: Create Workflow (`/admin/workflows/create`)

**Layout:** Form page with two-column layout on desktop (form left, preview right), single column on mobile.

**Employee Selection (D-03):**
- Search-first path: Text input with HTMX typeahead (`hx-get` on keyup with 300ms debounce) hitting existing search endpoint. Results shown in dropdown. Selecting an employee pre-fills name, email, and auto-detects job code from `EmployeeProfiles.ukg_job_code`.
- Manual entry path: Toggle link "Employee not in any system yet?" reveals simple form fields (name, email) and job code dropdown populated from `JobCode.get_active_job_codes()`.

**Job Code Dropdown:** `<select>` styled with Tailwind. Shows `job_code - title` format.

**Workflow Type:** Radio buttons -- "Onboarding" (default) / "Offboarding". Changing type updates the preview panel via HTMX.

**Preview Panel:** Read-only checklist preview showing items that will be generated. Rendered server-side via HTMX when job code + type are selected. Shows each item with its source ("From role mapping" or "Standard offboarding item").

**Primary CTA:** "Generate Checklist" button at bottom of form. `bg-blue-600 hover:bg-blue-700 text-white px-4 py-2 text-sm font-medium rounded-md`.

### Page: Workflow Detail (`/admin/workflows/<id>`)

**Layout:** Single-column detail page with header section and checklist section.

**Header:**
- Employee name (text-2xl font-semibold) + type badge (Onboarding/Offboarding)
- Employee email, job code, created date, created by
- Overall progress bar (same style as dashboard)
- Action buttons: "Export CSV" (secondary), "Delete Workflow" (destructive, requires confirmation)

**Checklist Section:**

Each checklist item is a card-like row:
```
bg-white border border-gray-200 rounded-md px-4 py-3 flex items-center gap-3
```

**Item states:**

| State | Visual |
|-------|--------|
| Pending | Unchecked circle icon (`fa-circle text-gray-300`), normal text |
| Completed | Green check circle (`fa-circle-check text-green-600`), text-gray-400 line-through, "Completed by [name] on [date]" subtitle |
| Skipped | Yellow dash circle (`fa-circle-minus text-yellow-500`), text-gray-400 italic, "Skipped by [name]: [reason]" subtitle |
| Overdue | Red left-border (`border-l-4 border-red-500`), red clock icon (`fa-clock text-red-500`) next to due date |

**Completion interaction (D-05 decision):** Individual checkboxes only. Each item has a checkbox that fires an HTMX POST on change. For a 4-5 person team with typical checklists of 5-15 items, bulk actions add unnecessary complexity. The checkbox triggers an inline confirmation for the item (no modal -- lightweight HTMX swap).

**Skip interaction (D-06):** Each item has a three-dot menu (`fa-ellipsis-vertical`) that reveals "Skip / N/A" option. Clicking opens an inline form (HTMX swap into the row) with a required textarea for the reason and "Confirm Skip" + "Cancel" buttons.

**Due date interaction (D-07):** Each item shows its due date (if set) as a small badge. Overdue items get `border-l-4 border-red-500` and the date badge turns `bg-red-100 text-red-800`.

**Grouping:** Items grouped by source -- "Role Mappings" section header, then "Standard Offboarding Items" section header (offboarding only). Section headers use `text-sm font-semibold text-gray-700 uppercase tracking-wider` with a horizontal rule.

### Page: Standard Offboarding Items (`/admin/workflows/offboarding-items`)

**Layout:** Simple list management page following existing admin patterns.

**List:** Sortable list (drag-and-drop via HTML5 drag API or manual up/down arrows) of standard offboarding items.

**Each item:** `bg-white border border-gray-200 rounded-md px-4 py-3 flex items-center justify-between`. Item text on left, action buttons (edit, delete, reorder arrows) on right.

**Add form:** Inline form at bottom of list -- text input + "Add Item" button. HTMX POST appends new item to list without page reload.

**Delete confirmation:** `window.confirm()` dialog matching existing compliance dashboard pattern. No modal needed for simple text items.

### Component: Admin Index Card (added to `admin/index.html`)

New card in the admin index grid:

```
bg-white rounded-lg shadow-sm border border-gray-200 p-6
  Icon: fa-list-check (blue-600)
  Title: "Workflows" (text-lg font-semibold)
  Description: "Onboarding and offboarding checklists" (text-sm text-gray-500)
  Link: url_for('admin.workflows_dashboard')
```

### Component: Profile Card "Start Onboarding" Button

Added to expanded profile cards in search results. Appears for admin users only.

```
inline-flex items-center px-3 py-1.5 border border-gray-300 shadow-sm text-xs font-medium
rounded-md text-gray-700 bg-white hover:bg-gray-50
```

Icon: `fa-clipboard-list mr-1.5`. Links to `/admin/workflows/create?employee_email={email}`.

## Interaction Patterns

### Checklist Generation (D-01 decision)

**Manual trigger approach.** Admin selects job code + employee, clicks "Generate Checklist". No template-first layer. Rationale: the job role mappings already serve as the template. Adding a template management layer for a 4-5 person team adds complexity without benefit. If customization is needed later, it maps cleanly to the v2 AUTO-01 scope.

### Item Scope (D-02 decision)

**Roles from mappings + standard offboarding items (no freeform per-workflow items).** Onboarding items come exclusively from `JobRoleMapping` entries (WKFL-01: "from job role mappings"). Offboarding additionally includes configurable standard items (D-10/D-11). Rationale: freeform items per workflow would require a text input UX that blurs the line between structured compliance tracking and ad-hoc task management. The standard offboarding items (managed via admin UI) cover the "equipment, orientation" category noted in D-02.

### Actionable Items (D-04 decision)

**Manual checkboxes only -- no execute buttons.** AUTO-01 is explicitly deferred to v2. Adding "Execute" buttons that call Phase 9 write operations would be indistinguishable from AUTO-01 in practice. Items describe what needs to be done; the admin performs the action in the relevant system (or uses existing WhoDis write operations from the search view), then marks the item complete.

### HTMX Interactions

| Interaction | Trigger | Target | Swap |
|-------------|---------|--------|------|
| Tab switch (Active/Completed) | Click tab | `#workflow-content` | innerHTML |
| Employee typeahead search | `keyup changed delay:300ms` | `#employee-results` | innerHTML |
| Job code change (preview update) | `change` on select | `#checklist-preview` | innerHTML |
| Mark item complete | `change` on checkbox | `#item-{id}` | outerHTML |
| Skip item (show form) | Click "Skip" | `#item-{id}` | outerHTML |
| Confirm skip (submit) | Click "Confirm Skip" | `#item-{id}` | outerHTML |
| Add standard offboarding item | Submit inline form | `#offboarding-items-list` | beforeend |
| Delete standard offboarding item | Click delete + confirm | `#item-{id}` | delete |
| Reorder offboarding items | Click arrow buttons | `#offboarding-items-list` | innerHTML |

### Loading States

All HTMX requests show the established loading indicator: `animate-spin rounded-full h-8 w-8 border-b-2 border-blue-600` with `htmx-indicator` class.

### Error Handling

Server errors on checklist operations return HTMX fragments with:
```
bg-red-50 border border-red-200 rounded-md p-4
  text-sm text-red-800: error message + retry instruction
```

## Accessibility Notes

- All form inputs have associated `<label>` elements
- Checkbox items use proper `<input type="checkbox">` with `aria-label` describing the checklist item
- Modal/dialog overlays use `role="dialog"` and `aria-modal="true"` (matching existing token revoke modal pattern)
- Tab navigation uses `aria-label="Workflow tabs"` on the `<nav>` element
- Progress bars use `role="progressbar"` with `aria-valuenow`, `aria-valuemin="0"`, `aria-valuemax="100"`
- Skip/N/A form textarea has `aria-required="true"`
- Color is never the sole indicator -- overdue items have both red border AND text label AND icon
