<objective>
Swap Flask-Limiter's rate-counter storage from in-memory to Redis via the
RATELIMIT_STORAGE_URI environment variable, closing the Phase 1 D-08 deviation
(SEC-03 shipped in-memory as a known deferral with this phase as the closure
point). A memory:// fallback preserves local dev and CI. A structured startup
warning fires when production mode is detected with memory:// storage so
misconfigured deploys are immediately visible in docker logs.

Purpose: Multi-worker gunicorn (WD-CONT-02) makes per-worker in-memory
rate counters incorrect — worker A and worker B each maintain independent
counters, so a user can exceed the rate limit by hitting different workers.
Redis provides a shared counter store visible to all workers.

Output:
- app/__init__.py — limiter module-level init updated; startup warning block added after limiter.init_app(app)
- requirements.txt — redis>=5,<6 added after Flask-Limiter line
- .env.sandcastle.example — RATELIMIT_STORAGE_URI section appended
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env var → Limiter constructor | RATELIMIT_STORAGE_URI is read from environment; a malformed URI causes Flask-Limiter to raise at startup (fail-fast) |
| container → Redis | The redis:// connection crosses the internal network; no auth on the SandCastle internal Redis service |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Information Disclosure | RATELIMIT_STORAGE_URI env var | mitigate | Value read via `os.environ.get` — never logged. The startup warning logs only the key name, not the value. Ensures Redis credentials are not echoed in docker logs. |
| T-03-01-02 | Tampering | In-memory fallback in production | mitigate | Production-mode startup warning (D-G2-02) makes misconfiguration immediately visible in docker logs; operator is forced to act before users are rate-limited incorrectly. |
| T-03-01-03 | Denial of Service | Unauthenticated Redis on internal network | accept | Redis is on the SandCastle internal network (WD-NET-01) — not reachable from the public proxy network. Risk is accepted; adding Redis auth is a SandCastle platform concern, not a Who-Dis concern. |
</threat_model>
