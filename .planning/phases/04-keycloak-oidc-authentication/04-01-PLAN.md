<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| upstream proxy → Flask app | The Keycloak-fronted reverse proxy terminates OIDC; Flask reads identity from `session["user"]` populated by the Authlib callback. The legacy Azure header is no longer trusted as an identity source. |
| Flask app → audit_log table | Audit writes must record the authenticated identity, not a header-derived string. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | Header-based identity reads (`X-MS-CLIENT-PRINCIPAL-NAME`) | mitigate | Remove all attribution reads of the header (Task 1, Task 2 Part A). After Keycloak fronting, the header is attacker-controllable from any client that can reach Flask directly; trusting it would let a caller forge audit identity. Replace with `g.user` (set by `@auth_required` from the OIDC session, which is signed via Flask `SECRET_KEY`). |
| T-04-02 | Repudiation | Audit log entries | mitigate | Regression test (Task 2 Part B) asserts audit rows show the OIDC-authenticated email after a write action. Prevents silent regression to "unknown" attribution that would let users deny actions. |
| T-04-03 | Information disclosure | Error logs / request dumps | mitigate | Retain `X-MS-CLIENT-PRINCIPAL-NAME` in `error_handler.py:239` `sensitive_headers` redaction list (D-G3-04). If a misconfigured proxy ever forwards the legacy header (e.g., during an Easy-Auth rollback test), it stays redacted from logs rather than appearing in plaintext error reports. Documented carve-out. |
| T-04-04 | Tampering | OIDC session cookie | accept | Out of scope for this plan — the Authlib/Flask session signing protections shipped in PR #25 are the existing control. No change to the cookie surface. |
| T-04-05 | Elevation of privilege | Role attribution drift | accept | `@require_role()` reads `g.role` (set alongside `g.user`); this plan does not touch role resolution. The role-resolution path was verified by audit. |
</threat_model>
