# Technology Stack

**Project:** WhoDis v4.0 — Platform Polish & Advanced Reporting
**Researched:** 2026-05-18

## Recommended Stack Additions

### No New Python Dependencies Required

The four v4.0 features are achievable with **zero** new pip dependencies. The existing stack (Flask 3.1.3, SQLAlchemy 2.0.45, msal 1.34.0, requests 2.33.0) already provides every integration point needed.

### Feature 1: SKU Friendly-Name Tooltips

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Existing Tailwind CSS | (already in templates) | Tooltip styling | Tailwind has built-in tooltip patterns via `group`/`group-hover`. No JS library needed. |
| Existing `SkuCatalogCache` | (already in codebase) | SKU GUID-to-name resolution | `app/services/sku_catalog_cache.py` already fetches `/subscribedSkus` and stores `name` (skuPartNumber) + `description` (displayName) in `external_service_data`. |

**New dependencies: NONE.** This is a Jinja2 template change + one new method on SkuCatalogCache.

**Confidence: HIGH** -- Verified by reading existing codebase.

### Feature 2: Schema Visualization (ER Diagrams)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Mermaid.js | 11.x (CDN) | Client-side ER diagram rendering | No server-side dependencies. Renders in browser from Mermaid syntax generated by SQLAlchemy metadata inspection. |
| Custom Mermaid generator | N/A | ~100 lines of Python using `db.metadata.sorted_tables` | No external dependency needed. |

**New dependencies: NONE** (Python side). Mermaid JS loaded from CDN on the admin page.

**Confidence: MEDIUM** -- SQLAlchemy inspect API is well-documented; Mermaid rendering is standard. Custom generator needs testing with the 20+ model schema.

### Feature 3: Multi-Stage Docker Build

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| `python:3.12-slim` | 3.12 | Both build and runtime base | Already using this. Multi-stage splits compile-time deps from runtime. |

**New dependencies: NONE.**

**Confidence: HIGH** -- Established pattern. ODBC driver copy is the only tricky part.

### Feature 4: Advanced M365 Reporting

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Existing `msal` + `requests` | 1.34.0 / 2.33.0 | Graph API calls | All usage report endpoints use same v1.0 `/reports/` pattern |
| Existing `ReportCache` model | (in codebase) | Cache report data | Same pattern used by Phase 8 license/security reports |
| Python `csv` stdlib | stdlib | Parse CSV responses | Graph usage reports return CSV via 302 redirect, not JSON |

**New dependencies: NONE.**

**Confidence: HIGH** -- Verified endpoint paths, permissions, and response formats against official docs.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| ER Diagrams | Custom Mermaid generator | eralchemy2 1.4.1 | **Deprecated** (May 2024). Requires system GraphViz. Adds Docker complexity. |
| ER Rendering | Mermaid.js (CDN) | GraphViz server-side PNG | Adds ~50MB system dep to Docker image, conflicts with Docker optimization goal |
| M365 Reports | Direct Graph API via requests | msgraph-sdk-python | Large dependency tree (~15 transitive deps). Project already has working GraphService. |
| Tooltips | Tailwind CSS `group-hover` | Tippy.js / Floating UI | External JS library for something Tailwind handles natively |
| Docker base | python:3.12-slim | Alpine | Alpine uses musl libc, causes issues with psycopg2, pyodbc, ODBC drivers |

## What NOT to Add

| Temptation | Why to Resist |
|------------|---------------|
| `msgraph-sdk-python` | ~15 transitive deps. `msal` + `requests` pattern works perfectly for CSV reports. |
| `graphviz` Python package | Requires system binary. Mermaid.js renders in browser with zero server deps. |
| `eralchemy2` | Deprecated. Don't add dead dependencies. |
| `pandas` for CSV parsing | stdlib `csv.DictReader` handles the small org-level CSVs. |
| New database tables for reports | `ReportCache` model already handles this with `report_type` + `cache_key`. |
